Getting started

Choose the right entrypoint.

There are two main ways to use Sidekar. Wrap an existing agent CLI in a PTY, or run Sidekar directly in standalone REPL mode.

1. Install the binary and skill

Run curl -fsSL https://sidekar.dev/install | sh. This installs the binary and runs sidekar install so the bundled skill is copied into detected agent skill directories.

2. Device login is optional

Run sidekar device login only when you need relay-backed features such as the web terminal, account session management, or account-backed encryption state.

3. PTY wrapper mode

Use sidekar <agent> [args] to wrap Claude Code, Codex, Gemini CLI, or another agent CLI. This is the path that gives you bus registration, relay, and browser isolation around the external agent process.

4. REPL mode

Use sidekar repl -c <credential> -m <model> to run Sidekar as the agent. This path has session persistence, slash commands, model credentials, and a single bash execution tool.

Overview

Product pillars vs command groups.

Sidekar’s public framing is four pillars. The CLI is more granular because discoverability matters at the command level.

Product-level framing

  • Browser: one surface covering launch/connect, page reading, interaction, and data capture.
  • Desktop: native macOS automation outside the browser.
  • Agent: coordination, continuity, and local context.
  • Background: reactive monitors and scheduled jobs.

Current CLI groups

  • Browser, Page, Interact, Data for the browser surface
  • Desktop for native app automation
  • Agent for bus, memory, tasks, repo, compact, monitor
  • Jobs for cron and loop
  • Account for device, session, KV, TOTP
  • System for daemon, config, feedback, event, install, uninstall, update, skill, ext
Browser

Launch, read, interact, inspect.

The browser pillar spans four CLI groups: Browser, Page, Interact, and Data. That split is for command discoverability, not product marketing.

Browser commands

Use launch, connect, navigate, tabs, tab, new-tab, close, frames, and frame to manage sessions, tabs, and navigation.

Page commands

Use read, text, dom, ax-tree, observe, find, resolve, screenshot, pdf, search, and read-urls to perceive page state.

Interact commands

Use click, type, fill, press, scroll, upload, drag, wait-for, wait-for-nav, and eval to drive actual UI flows.

Data commands

Use cookies, console, network, block, viewport, download, storage, service-workers, and security to inspect and control runtime state.

Desktop

Work outside the browser.

Desktop automation is for the parts of real workflows that CDP cannot touch: native app windows, OS dialogs, screenshots, app switching, and local UI interaction.

Capabilities

Use the desktop subcommands to inspect apps and windows, click elements, type, paste, press keys, take screenshots, launch apps, activate them, and quit them.

When it matters

Use desktop automation when the work crosses browser boundaries: login dialogs, Finder interactions, app-specific prompts, or native tooling around browser-based workflows.

Agent

Keep coordination and context local.

The agent pillar is where Sidekar stops being “just browser automation.” This is the local coordination and continuity layer around agent work.

Bus

bus gives you local agent discovery, directed messages, replies, handoffs, and request tracking on the local SQLite-backed broker.

Memory and tasks

memory stores durable project-scoped facts. tasks stores local task dependencies so the next unblocked work is queryable instead of living only in the chat transcript.

Repo and compaction

repo packs repository context, summarizes changes, and discovers project actions. compact reduces noisy command output for agent use.

Agent sessions and monitor

agent-sessions gives local session history. monitor watches tabs for background changes and routes notifications back through the bus.

Background

Keep work running after the prompt.

Background automation covers reactive monitoring and scheduled work. The important boundary is that these are host-owned runtime features, not just prompt conventions.

Reactive work

  • monitor watches tabs for title or favicon changes and reports back on the bus.
  • Use it when a long-lived browser state matters more than one immediate page read.

Scheduled work

  • cron manages scheduled jobs.
  • loop is the recurring prompt shortcut.
  • Use them for periodic checks, recurring prompts, or background tool runs.
REPL

Run Sidekar as the agent.

sidekar repl is not a product pillar. It is a top-level access mode with its own provider setup, session persistence, slash commands, and standalone agent loop.

Credentials and models

Use sidekar repl login <provider>, sidekar repl credentials, and sidekar repl models -c <credential> to manage provider access. Credential prefixes determine provider family.

Interactive workflow

Use /credential, /model, /sessions, /resume, and /compact inside the REPL. Session history and mini-line history are persisted in SQLite.

Tool surface

The model-visible tool surface is intentionally minimal. The current REPL exposes one tool: bash. Sidekar capabilities are reached by running the CLI from that tool.

Prompt boundary

The REPL builds its own system prompt with capability guidance, environment context, memory brief injection, and explicit safety rules against prompt injection, secrets exfiltration, and destructive actions.

Account and system

The supporting runtime layer.

These commands are not separate product pillars, but they make the rest of Sidekar usable as a local runtime.

Account

Use device for machine authentication, session for active account sessions, kv for encrypted key-value storage, and totp for stored one-time-password secrets.

System

Use daemon, config, event, update, install, uninstall, skill, and ext to manage the host runtime and supporting surfaces.

Extension

One browser access surface, not a fifth product pillar.

The Chrome extension is for the harder browser cases and for access to the user’s everyday Chrome profile. It complements CDP; it does not replace the browser pillar.

Use the extension when you need your signed-in browser state or when an in-page bridge is more reliable on complex SPAs. Skip it when plain CDP is enough. The detailed install, permission, and trust model lives on the dedicated extension page.
