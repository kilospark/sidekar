Four pillars

One sidecar. Four surfaces agents can actually use.

Sidekar gives agent CLIs four concrete capabilities: browser automation, desktop automation, a shared bus for coordination, and background jobs that keep running after the prompt ends.

01

Browser automation

Raw Chrome DevTools Protocol plus an optional Chrome extension for your everyday profile. Read, click, type, screenshot, capture network, and keep real logged-in context when the job needs it.

02

Desktop automation

macOS Accessibility API support for windows, menus, dialogs, and native app controls that live outside the browser.

03

Inter-agent bus

Find peers, hand off work, send results, and coordinate over a local SQLite broker, with remote delivery available through the relay when sessions span machines.

04

Background automation

Monitors and scheduled jobs keep operating after the agent returns control, then feed the result back through the same bus.

Chrome extension

Use your normal Chrome profile when isolated CDP is not enough.

The extension is the local bridge into your everyday browser. It is for flows where the agent needs the tabs, cookies, and logins you already use instead of a separate automation profile.

Use the Chrome you already work in, with a trust model you can actually explain.

Sidekar’s extension connects to a local bridge on 127.0.0.1. It signs in separately from the CLI so the bridge can verify the browser user and the CLI user belong to the same account.

Local by default The extension only talks to the local machine. It exists to let Sidekar interact with tabs in your real Chrome profile.
Why native messaging exists It starts or discovers the local Sidekar bridge and proves the extension is the official one, not an arbitrary page probing localhost.
Why host access is broad Agents need to automate whatever site you explicitly direct them to. The extension is inactive until your local Sidekar command asks it to do work.

Workflows

Work that crosses real boundaries.

Sidekar matters when an agent has to move from code to browser, from browser to desktop, or from one agent to another without losing state.

Ship and verify

One agent edits the code, another opens the actual app in Chrome, and a third watches staging. The bus passes results back without you manually copying context between terminals.

  • sidekar codex builds the change
  • sidekar ext read checks the real page
  • bus_send hands off to QA

Complete login flows

Agents can read credentials from encrypted local storage, generate TOTP codes, and finish browser login flows end to end instead of stopping at “please do this step manually.”

  • Encrypted KV for secrets
  • TOTP generation on demand
  • Real browser and desktop surfaces

Watch the background

Keep long-running checks alive after the prompt: watch a dashboard, poll a page, or schedule work and route the result back into the same agent bus.

  • Monitors for reactive alerts
  • Cron for recurring checks
  • Results delivered through the bus
Security and trust

Make the trust model obvious.

Browser automation, desktop automation, local bus delivery, secrets, and TOTP all stay on your machine. sidekar.dev handles authentication, dashboards, device management, and relay sessions.

What stays local

Browser automation, desktop automation, local bus delivery, stored secrets, and TOTP generation happen on your machine. The extension talks to a local bridge on 127.0.0.1. It does not browse on its own and it does not act until your local Sidekar process tells it to.

Remote features exist for authentication, dashboards, and relay sessions. They are a separate layer, and this site spells out that boundary instead of hiding it in docs.
